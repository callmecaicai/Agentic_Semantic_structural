# Document Memory Protocol

本文档规定语义化项目中的 Markdown 如何写、何时同步、何时不治理。

核心原则：

> md 是项目理解的长期记忆层。  
> 但文档维护必须服从任务本身。

---

## 1. Markdown 语义单元

单篇 md 遵守：

```text
路径承担身份。
文件名承担对象。
标题承担问题。
首段承担定性。
二级标题承担功能切片。
正文承担内容。
必要标注承担防误用。
```

不要让每篇文档都重复复杂元数据。

路径已经说明它的大类身份。

---

## 2. 文件名和标题

文件名回答：

> 这篇文档围绕哪个对象？

标题回答：

> 这篇文档要回答什么问题？

避免：

```text
notes.md
summary.md
misc.md
final.md
temp.md
```

推荐：

```text
change_decoder.md
dataset_contract.md
exp_042_decoder_ablation.md
adr_003_sota_protocol.md
```

---

## 3. 正文写法

正文要区分：

- 当前事实；
- 稳定规则；
- 当前假设；
- 实验观察；
- 风险边界；
- 验证方式；
- 后续回流点。

推荐写法：

```text
Rule:
Assumption:
Observation:
Risk:
Verify with:
Related run:
```

不要把猜测写成事实。

不要把历史实验写成当前结论。

---

## 4. 文档类型

### 核心文档

写当前有效理解。

少量、稳定、持续更新。

### 模块文档

写模块职责、接口、输入输出、不变量、验证方式和失败边界。

### 实验文档

写一次历史实验。

原则上冻结，只追加 correction 或 follow-up。

### 决策文档

写为什么这样选，以及什么结果会推翻这个选择。

### 失败文档

写具有复现性和结构警示价值的失败模式。

---

## 5. 最小同步规则

文档只在任务自然需要时同步。

### 模块同步

代码修改改变模块接口、配置、行为、验证方式或失败边界时，同步模块文档。

局部 bug 修复、格式调整、内部优化，不改变模块语义时，不同步。

### 实验同步

正式实验必须生成或更新实验文档。

实验文档记录当时状态，不自动改写核心理解。

### 用户触发同步

核心文档、架构文档、决策文档和失败模式文档默认不由 agent 自动更新。

只有用户明确要求，或当前任务明确包含文档更新时才修改。

---

## 6. 不主动治理

agent 不应默认：

- 扫描全项目找冲突；
- 自动拆分文档；
- 自动回流实验结论；
- 自动修改核心理解；
- 自动废弃旧文档；
- 自动重构目录结构。

遇到自然冲突时，只指出冲突并暂停高风险修改。

---

## 7. 最高原则

> 文档降低长期行动成本。  
> 文档不制造额外维护负担。  
> 实验文档保存历史。  
> 核心文档表达当前理解。  
> 高权威文档等待裁决。  
> 重要结果可以提出回流，但不能静默篡改项目理解。
